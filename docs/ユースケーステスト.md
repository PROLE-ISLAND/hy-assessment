あなたはソフトウェア品質設計者。目的は「Playwrightを使ったユースケース（Gold/E2E）テスト設計」を GitHub Wiki にまとめ、誰（人間/AI）が開発しても同じルールで動く開発体制を作ること。

# ゴール
- GitHub Wiki に、Playwrightベースのユースケーステスト（Gold/E2E）の設計思想・MECEな洗い出し方法・選別（Triage）・テスト仕様テンプレ（GWT）・運用ルール（DoD/CI）をまとめる。
- 「ユースケースは機能単位ではなく Role（主体）× Outcome（価値/完了）で定義する」「フロントエンドは中心軸ではなく入口（Channel）」「プロセスはシナリオ（手順）としてGWTに落とす」を明確にする。
- Goldは最小本数（例: 5〜10本）で「事業が死んでない証拠」に限定し、詳細はBronze/Silver（Unit/Integration）へ逃がす方針を明文化する。

# 作ってほしいWikiページ（ページ名も含めて提案してOK）
最低でも以下を作成して。各ページは短く、判断基準・禁止事項・例外条件を中心に書く（HOWTOの細部や長文手順は避ける）。

1) Home（目次と全体像）
2) Gold Test Charter（ゴールドの目的・本数上限・採用基準・失敗時の扱い）
3) Usecase Definition (Role × Outcome)（ユースケースの定義、機能単位との違い、Channel/Scenarioの位置づけ）
4) Gold Usecase Generation (MECE)（Role/Outcome/Channelの掛け算→候補生成→削減の方法）
5) Gold Triage Scoring（Impact/Frequency/Detectability/Recovery Costのスコアリング、閾値例、採用/却下ルール）
6) Gold Spec Template (GWT)（Given/When/Thenテンプレ、assertの原則、見て良い/見てはいけない）
7) Playwright Design Rules（セレクタ方針、待機の原則、テストデータ、ネットワーク観測、トレース/動画/スクショ運用）
8) Anti-patterns（E2E地獄を起こす典型例と回避策：全画面E2E、実装寄りassert、スナップショット乱用など）
9) DoD & CI Gate（Bronze/Silver/GoldのDoDレベル、CIでの強制、例外フロー）

# 重要な内容（必ず入れる）
- Gold/E2Eは「事業が成立している証拠」：フローの完了を主張する。細部（文言/ボタン色/並び順など）を見ない。
- 「ボタン押下で正しい挙動」は原則Bronze/Silverで検知されるべき。Goldは“流れの破綻”だけを見る。
- テストの価値判断レンズ（最低限）：(1)壊れたら困るか (2)実装ではなく振る舞いを見てるか (3)嘘をついて通れないか (4)落ちた理由を1文で説明できるか (5)無いと何を諦めるか
- PlaywrightはE2E専用ではなく「UIを入口に統合的に観測できる装置」。ただしGoldは最小に保つ。
- 例外（守らない領域）を明文化して、過剰網羅を抑制する。
- “Single Source of Truth”はWikiで、CLAUDE.md/AGENTS.md/Copilot instructionsはWiki参照の入口にする（ここは簡潔に触れるだけでOK）。

# 出力形式
- まず「Wikiのページ一覧（ページ名＋1行説明）」を出す。
- 次に、各ページのドラフト本文を Markdown で出す（見出し構造、箇条書き中心）。
- それぞれのページ内に、必要ならテンプレ（GWT、スコア表、チェックリスト）を含める。
- 最後に「このWikiをCI/PRテンプレと接続する最小手順（やること3〜5個）」を書いて。

# 追加の前提
- 具体プロダクトの種類はECでもSaaSでも成立するよう抽象化して書くが、例としてECの「検索→購入完了」ユースケースを1本だけサンプルでGWT化して入れる。
- 文章は断定的に。運用に落ちるルールとして書く（曖昧な表現を避ける）。
